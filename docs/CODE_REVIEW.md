# Code Review 规范（Puer 方法论）

本文档定义 CI 自动 review 和本地 review 的共同规范。
由 `claude-code-action` 通过 CLAUDE.md 自动加载。

## 工程思维原则

1. **诊断先于治疗** — 先看全景理解意图，再逐条审查
2. **事实验证** — 每条 finding 用 `cat`/`grep`/`git log` 验证实际代码，不凭 diff 猜测
3. **不确定 = 停** — confidence < 85% 的不报告，对框架行为不确定的不下结论
4. **每个细节都有意图** — 不报风格偏好，不建议"可以考虑"级别的改动

## 审查流程

1. 获取变更全景（`gh pr diff` 或 `git diff`）
2. 对每个变更文件，读上下文再判断（不仅看 diff）
3. 用 `git log` 确认是新引入的问题还是已有的
4. 每条 finding 分类：
   - **真问题**（confidence >= 85%）→ 报告
   - **误报** → 静默丢弃
   - **有效建议但非本 PR 范围** → 简要提及，不阻塞
   - **不确定** → 不报告

## 审查维度（按优先级）

| 类别         | 关注点                                      |
| ------------ | ------------------------------------------- |
| 安全性       | 凭证泄露、注入、不安全操作                  |
| 逻辑正确性   | 行为不正确、缺少边界处理、竞态条件          |
| 资源管理     | 未关闭的句柄、缺少清理、内存泄漏            |
| API 与兼容性 | 影响消费者的接口变更、破坏性修改            |
| 代码质量     | 静默失败、错误被吞没、fallback 隐藏真实问题 |

## NEVER

- NEVER 报告风格/格式问题（ESLint + Prettier 处理）
- NEVER 猜测框架行为——用代码或文档验证
- NEVER 将与代码库一致的现有模式报为问题
- NEVER 建议"锦上添花"的改进——只报告真实问题
- NEVER 报告 CHANGELOG.md 或自动生成文件的问题

## 输出格式

```markdown
## Automated Code Review

> ⏱️ Review cost Xm Xs

**概要**: 一句话描述本 PR 的核心改动意图

### 检查清单

| 类别         | 状态     | 备注     |
| ------------ | -------- | -------- |
| 安全性       | ✅       | 简要说明 |
| 逻辑正确性   | ✅/⚠️/❌ | 简要说明 |
| 资源管理     | ✅       | 简要说明 |
| API 与兼容性 | ✅/⚠️    | 简要说明 |
| 代码质量     | ✅/⚠️    | 简要说明 |

说明: ✅ 通过 | ⚠️ 警告 | ❌ 未通过 | ➖ 不适用

### 🚫 阻塞问题

- `file/path:line` - 问题描述 + 修复建议（Claude）

### ⚠️ 警告

- `file/path:line` - 问题描述 + 修复建议（Claude）

### 💡 建议

- `file/path:line` - 建议描述（Claude）

---

✅ 已审查 N 个文件，M 处变更
```

### 格式规则

- 每个分类无内容时写"无。"
- 每条必须包含 `file/path:line` 引用
- 每条 1-2 句话，末尾标注来源 `（Claude）`
- 使用中文（简体）
- 阻塞问题和警告同时添加 inline comment
