# Maker CLI-First Rework Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to continue this
> branch task-by-task. This plan is also the resume checkpoint after context compaction. Keep checkbox
> status current when you change code.

**Goal:** Move Maker onboarding to CLI, reduce Maker MCP to the development-loop surface, and merge
submit/push/build into one user-facing operation.

**Architecture:** CLI owns one-time local setup: Git/PAT/TapTap token/app selection/dev-kit/clone/MCP
client config. MCP owns only runtime status and the high-frequency sync+build loop. Internal push and
build phases remain separate so push failures, remote rejections, and conflicts can be recovered by the
local Agent before remote build starts.

**Tech Stack:** TypeScript, Node.js built-ins, MCP SDK, Jest. Avoid new runtime dependencies in this
iteration. Tests are focused on changed contracts only; broad end-to-end smoke is owned by local manual
Maker environments.

## Key Decisions

1. CLI is the onboarding owner.
   `taptap-maker init` performs Git check, PAT save, TapTap token exchange, app selection, dev-kit
   preparation, clone, MCP config install, and verification.

2. MCP is intentionally small.
   Keep `maker://status`, `maker_status_lite`, and `maker_build_current_directory`. Do not add
   onboarding tools back just because they are easy to expose.

3. `maker_build_current_directory` is the single user-facing submit/build tool.
   User wording like "提交", "push", "构建", "预览", "跑一下", and "验证一下" maps here. It commits
   when needed, pushes to Maker remote, then builds. If push fails, remote build is not started.

4. Public MCP status must not mutate local setup.
   `maker://status` may refresh TapTap token from existing PAT, but it should not download or restore
   dev-kit files. Dev-kit repair belongs to `taptap-maker dev-kit update`.

5. Windows-first behavior is a product requirement.
   Generated MCP config should use `npx.cmd` on Windows. Git guidance should prioritize Git for
   Windows and PATH visibility from third-party tools.

6. MCP reload cannot be fully bypassed from inside an already running Agent.
   CLI can install config and continue PAT/app/clone now, but newly installed MCP tools normally appear
   after client reload, MCP refresh, or a new session. Docs should state this plainly.
